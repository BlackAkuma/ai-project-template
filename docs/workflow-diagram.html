Chapter 01
สถาปัตยกรรม 3 ชั้น
1
core/ — รากฐานของทุกโปรเจ็กต์

2

3
platforms/claude-code/ — deep integration

Chapter 02
Session Loop
1
Bootstrap — ครั้งแรกเท่านั้น
AI อ่าน cross-project-memory ก่อน (ถ้ามี) แล้วถามภาษาและประเภทโปรเจ็กต์ ถ้าเป็น game → โหลด skills/game/ เพิ่มอัตโนมัติ จากนั้นสร้าง doc/ structure ทั้งหมด
2
Session Start — 3 ไฟล์หลัก
อ่าน AI-CONTEXT block ของ work-status + work-log + task-board แค่ 3 ไฟล์นี้ AI รู้ทุกอย่างที่ต้องรู้ จากนั้นตาม read-more hints โหลดเฉพาะส่วนที่ task นี้ต้องการ
3
ทำงาน — ภายใต้ protocol ทั้งหมด
AI ทำงานตาม decision protocol (chapter 04), compliance check C-01–C-14, entity lifecycle ถ้าเจอสิ่งที่ไม่รู้ → ใช้ scenario ที่เหมาะสม ไม่เดา ไม่ข้ามขั้น
4
Session End — ห้ามข้าม
อัปเดต work-status, work-log, task-board ทุกครั้ง ตรวจว่ามี entity ที่เปลี่ยน status ไหม ถ้าใช้ multi-tool เขียน agent diary ของ tool นี้ ถ้ามี pattern ข้ามโปรเจ็กต์ → ถามผู้ใช้ก่อนเสมอ
💡
3 ไฟล์ = เพียงพอ — ทุก session อ่านแค่ AI-CONTEXT 3 ไฟล์ ใช้ token น้อย ได้ context ครบ ไม่ต้อง scan ทั้งโปรเจ็กต์

Chapter 03
Task Lifecycle
1
todo → Definition of Ready
Task ที่ผ่าน Definition of Ready จะต้องมี: source doc reference, scope ชัดเจน, และ acceptance criteria ที่ตรวจสอบได้ ถ้ายังไม่มี source ref → ติด NEEDS SOURCE VALIDATION ทันที
2
design_validate — ตรวจก่อนลงมือ
AI ตรวจว่า source doc confirm scope นี้หรือไม่ ถ้า source doc ขัดแย้งกัน หรือขาดข้อมูล → BLOCKED CONFLICT ห้าม AI เลือกเองเงียบ ๆ ต้องให้ human ตัดสินใจ
3
in_progress — ทำงานภายใต้ protocol
AI ทำเฉพาะ scope ที่ design_validate ผ่านแล้ว ถ้าเจอ scope creep → log เป็น task ใหม่ ถ้าเจอบั๊กนอก scope → log FOUND-IN-PASSING แล้วทำงานเดิมต่อ
4
review → Definition of Done
Definition of Done ต้องครบ: code ตรงกับ source doc version ที่ใช้, compliance check ผ่าน C-01–C-14, task-board อัปเดต, work-log บันทึกแล้ว
5
BLOCKED — หยุดอย่างมีระเบียบ
BLOCKED ไม่ใช่ความล้มเหลว — เป็นกลไกป้องกัน AI เดา BLOCKED มี tags บอกสาเหตุชัด human เห็นปัญหาและ resolve ได้ทันที จากนั้น task กลับไป todo และเริ่มใหม่
💡
หยุดมีราคาถูกกว่าเดา — task ที่ BLOCKED พร้อม description ชัดแก้ได้เร็ว task ที่ AI เดาแล้วผิดแก้ยากกว่ามาก

Chapter 04
Decision Protocol
1
3-Question Check ก่อนทุกการกระทำ
1) รู้ผลลัพธ์ที่คาดหวังไหม? 2) ตรงกับ source docs ไหม? 3) กลับคืนได้ถ้าผิดไหม? ถ้าทั้ง 3 ใช่ → ดำเนินการต่อ ถ้าข้อใดข้อหนึ่งไม่ใช่ → ใช้ scenario ที่เหมาะสม
2
Scenario A–D — ปัญหาทั่วไป
A: task คลุมเครือ → หา pattern เดิมก่อน ถ้าไม่มี BLOCKED · B: code ไม่ตรง source → extension doc + NEEDS HUMAN · C: scope creep → log task ใหม่ ไม่ทำเกิน · D: เจอบั๊ก → FOUND-IN-PASSING ไม่แก้เดี๋ยวนั้น
3
Scenario E–H — ปัญหาที่ซับซ้อน
E: requirements ขัดแย้ง → BLOCKED CONFLICT ไม่เลือกเอง · F: ไม่มีข้อมูลในเลย → placeholder ชัดเจน ห้ามเดา · G: context ใกล้เต็ม → บันทึก checkpoint ก่อนหมด · H: gap ใน source ref → ขนาดเล็กแก้ inline ใหญ่ block ทั้งหมด
4
Escalation Levels — 3 ระดับ
L1 Log and Continue — stake ต่ำ reversible ได้ บันทึกแล้วทำต่อ · L2 Block and Flag — stake กลาง mark BLOCKED อัปเดต work-status รอ human · L3 Stop Session — stake สูง (data loss, security, irreversible) หยุดทันที log ชัดเจน
💡
"Do less, document more" — หลักการสำคัญที่สุดของระบบ การหยุดที่ document ไว้ชัดดีกว่าการตัดสินใจผิดแบบเงียบ ๆ เสมอ
NEVER AI ห้ามเปลี่ยน requirements เชิงธุรกิจเอง — human only
NEVER AI ห้ามทำ production operations โดยไม่มี explicit permission

Chapter 05
Memory System
1
Layer 1 — Entity Memory
AI ไม่รู้ว่า tech ที่เคยเลือกไว้ยังใช้อยู่ไหม entity-register.md แก้ปัญหานี้ — track ทุก tech decision พร้อม status (active / deprecated / superseded) และช่วงเวลา เจอ [ENTITY:deprecated] ในไฟล์ไหน → ตรวจ register ก่อนใช้เสมอ (C-14)
2
Layer 2 — Context Routing
AI อ่าน AI-CONTEXT 3 ไฟล์แล้วยังไม่รู้ว่าต้องอ่านอะไรเพิ่ม read-more hints ใน work-status บอกชัดว่าถ้าต้องการ context เรื่องอะไรให้ไปอ่านที่ไหน โหลดเฉพาะส่วนที่ task ปัจจุบันต้องการ ประหยัด token
3
Layer 3 — Agent Isolation
ถ้าใช้หลาย AI tool บนโปรเจ็กต์เดียว work-log จะปนกัน agents/ folder แก้ปัญหานี้ — แต่ละ tool เขียน diary ของตัวเอง ห้าม cross-write session เริ่ม → อ่าน diary ของ tool ที่ใช้อยู่ก่อน
4
Layer 3 — Cross-Project Memory
pattern และ lesson ที่เรียนรู้ไม่ตกหายเมื่อโปรเจ็กต์จบ cross-project-memory.md อยู่นอก git ระดับ user workspace AI อ่านตอน bootstrap ใหม่ เขียนเพิ่มได้เฉพาะ pattern ที่ proven แล้ว และต้องขออนุญาตผู้ใช้ก่อนเสมอ
💡
Memory ≠ Context Window — memory system ทำงานข้ามเวลา ข้ามโปรเจ็กต์ และข้ามเครื่องมือ context window หมดได้ แต่ระบบ memory ยังอยู่
C-14 เจอ [ENTITY:deprecated] ต้องตรวจ entity-register ก่อนใช้ข้อมูลนั้น
Rule ห้ามเขียน cross-project-memory โดยไม่ได้รับอนุญาตจากผู้ใช้
