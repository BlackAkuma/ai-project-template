AI ทำงานกับโปรเจ็กต์ได้ดี — แต่ไม่ได้ตลอด

ทุกครั้งที่เปิด session ใหม่ AI เริ่มจากศูนย์ ไม่รู้ว่าเคยตัดสินใจอะไรไว้ ไม่รู้ว่างานอยู่จุดไหน ไม่รู้กฎที่ทีมตกลงกันไว้ ปัญหานี้ระบบนี้แก้ได้

❌ ไม่มีระบบ — ปัญหาที่เกิดทุกวัน

Decision Protocol — ไม่เดา ไม่แข็งทื่อ

ระบบมี 11 scenario สำหรับทุกสถานการณ์ที่ AI อาจเจอ แต่ละ scenario บอกชัดว่าต้องทำอะไร ผลที่ได้คือ AI ที่หยุดอย่างมีระเบียบแทนการเดาผิด

01
รู้ผลลัพธ์ที่คาดหวังไหม?
ถ้าทำแล้ว ผลที่ออกมาจะเป็นอะไร — รู้ชัดพอที่จะตรวจสอบได้ไหม
02
ตรงกับ source docs ไหม?
สิ่งที่จะทำมี requirement รองรับใน source document หรือ ADR ที่ accept แล้วไหม
03
Reversible ถ้าผิด?
ถ้าตัดสินใจผิด กลับแก้ได้ไหม หรือมีผลกระทบที่ undo ไม่ได้
ถ้าผ่าน 3 คำถาม → ดำเนินการต่อ / ถ้าไม่ → ดูสถานการณ์ที่ตรงกัน
A
Task description คลุมเครือ — ไม่รู้จะเริ่มจากตรงไหน
เกิดเมื่อ: task บน task-board ขาด acceptance criteria หรือ scope ไม่ชัด
L2 — Block ▼
AI ทำอะไร: ก่อนอื่นหา task ที่คล้ายกันที่ผ่านมาแล้ว ดู pattern ที่เคยใช้ ถ้าไม่มี → mark task ว่า [BLOCKED] พร้อมระบุชัดว่าขาดข้อมูลอะไร เช่น "ต้องการ: scope ของ payment flow ว่าครอบคลุม refund ด้วยไหม" แล้วอัปเดต work-status
📋 mark BLOCKED + อธิบาย → รอ human ตอบ
B
Code ปัจจุบันขัดแย้งกับ source doc
เกิดเมื่อ: อ่าน source doc แล้วเห็นว่า requirement ต่างจากที่ implement ไปแล้ว
L2 — Block ▼
AI ทำอะไร: ห้าม resolve เงียบ ๆ เองเลือกข้างหนึ่ง สร้าง extension doc บันทึกความขัดแย้งชัดเจน: "source doc v1.2 บอก X แต่ code ทำ Y" แล้ว mark task ว่า [NEEDS HUMAN DECISION] อัปเดต work-status ให้เห็น
📄 extension doc + NEEDS HUMAN → รอ human ตัดสินใจ
C
Scope Creep — เจอสิ่งที่ "น่าทำ" นอก scope
เกิดเมื่อ: ระหว่างทำ task A เจอว่า "ถ้าแก้ B ด้วยตอนนี้ก็น่าจะดี"
L1 — Log ▼
AI ทำอะไร: ทำ task เดิมต่อตามที่ scope กำหนด ห้ามออกนอก scope เอง สร้าง task ใหม่ใน task-board สำหรับ B พร้อม tag [NEEDS SOURCE VALIDATION] ถ้ายังไม่มี source ref รองรับ log ใน work-log ว่าเจออะไรและทำอะไร
📋 log task ใหม่ → ทำงานเดิมต่อ
D
พบ bug นอก scope ระหว่างทำงาน
เกิดเมื่อ: เห็น bug ที่ไม่เกี่ยวกับ task ปัจจุบันเลย แต่รู้ว่ามีปัญหา
L1 — Log ▼
AI ทำอะไร: ห้ามแก้ตอนนี้เด็ดขาด log มันเป็น task ใหม่ใน task-board พร้อม tag [FOUND-IN-PASSING] ระบุ: "พบระหว่างทำ T-023, ยังไม่กระทบ production แต่ควรแก้ใน sprint ถัดไป" แล้วกลับมาทำ task เดิมต่อ
📋 log FOUND-IN-PASSING → กลับทำงานเดิม
E
Requirements สองอันขัดแย้งกัน
เกิดเมื่อ: source doc A บอกว่าต้องทำ X แต่ source doc B (หรือ ADR) บอกว่าทำ Y
L3 — Stop ▼
AI ทำอะไร: หยุดทันที ห้ามเลือกข้างเองแม้แต่ข้างเดียว สร้าง extension doc บันทึก conflict ชัดเจนพร้อม reference ทั้งสองฝั่ง mark task ว่า [BLOCKED: CONFLICT] อัปเดต work-status ให้ escalate ออกมาชัดเจน human ต้องเลือกเองเท่านั้น
🛑 BLOCKED CONFLICT → ต้องการ human decision เท่านั้น
F
ไม่มีข้อมูลในเอกสารใดเลย
เกิดเมื่อ: ต้องตัดสินใจเรื่องนึงแต่ไม่มีใน source doc, ADR, หรือ way-of-work เลย
L2 — Block ▼
AI ทำอะไร: ใส่ placeholder ชัดเจนในจุดที่ขาดข้อมูล เช่น <NEEDS_CLARIFICATION: ต้องการ timeout policy สำหรับ payment session> ห้ามเดาหรือใส่ค่า default เองโดยไม่บอก log ใน work-status ว่าขาด context อะไรบ้าง
📌 placeholder + log gap → รอ human ให้ข้อมูล
G
Context window กำลังจะเต็ม
เกิดเมื่อ: ทำงานมานานและ context เริ่มหมด ยังทำไม่เสร็จ
L1 — Log ▼
AI ทำอะไร: บันทึก checkpoint ก่อนที่ context จะหมด เขียน progress note ใน work-log ว่าทำถึงตรงไหน mark task ว่า [IN_PROGRESS: checkpoint saved] ระบุชัดว่า "หยุดตรงนี้เพราะ context เต็ม ขั้นถัดไปคือ X" session ถัดไปรับต่อได้ทันที
📍 checkpoint saved → session ถัดไปรับต่อ
Level 1 — Log and Continue
Stake ต่ำ · Reversible
บันทึกสิ่งที่พบ แล้วทำงานต่อได้ ผลกระทบต่ำ แก้ได้ทีหลัง
Level 2 — Block and Flag
Stake กลาง · ต้องการ human
mark BLOCKED อัปเดต work-status รอ human ตัดสินใจก่อนทำต่อ
Level 3 — Stop Session
Stake สูง · Irreversible
หยุดทันที log ชัดเจน รอ human สั่ง resume เท่านั้น ใช้เมื่อเกี่ยวกับ data / security / production
จำข้ามเวลา ข้ามโปรเจ็กต์

Memory System — 3 ชั้น ไม่ขึ้นกับ context window

ปัญหาของ AI คือ memory หายเมื่อ session จบ ระบบนี้แก้ด้วย 3 ชั้น: entity ที่ track สถานะ, context routing ที่โหลดเฉพาะสิ่งที่ต้องการ, และ memory ที่ข้ามโปรเจ็กต์ได้

🏷
Layer 1 — Phase 1
Entity Memory
AI รู้ว่า tech ที่เลือกไว้ยังใช้อยู่หรือ deprecated แล้ว
entity-register.md
📋 entity-register.md
ไฟล์เดียวที่ track ทุก tech decision พร้อม status active / deprecated / superseded / proposed รวมถึงวันที่เริ่มใช้, วันที่เลิกใช้, และ ADR ที่อ้างอิง

ตัวอย่าง: JWT · auth · active · 2026-01 · ADR-001 และ Redux · state · deprecated · until 2026-03 · ADR-005
กฎ C-14: เจอ [ENTITY:deprecated] → ตรวจ register ก่อนใช้เสมอ
🔖 Entity Lifecycle Tags
ใส่ tag ใน ADR และ task ที่อ้างถึง entity นั้น:

[ENTITY:deprecated:Redux] — อ่านเจอต้องตรวจ register
[ENTITY:superseded:JWT→OAuth2] — ถูกแทนที่ด้วยอะไร
[ENTITY:proposed:GraphQL] — ยังไม่ได้ตัดสินใจ

validate-commit.sh hook จะ warn อัตโนมัติถ้า commit มี reference ถึง entity ที่ deprecated
